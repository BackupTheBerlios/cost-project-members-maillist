> Dear Teunis and colleagues,

> We're almost there! but we need to refine a little.

> Starting from Option 1, I would keep the "P" in Sampling type, and 
> Partial in HH.CatchRegistration means only one fraction (D/L) has been 
> sampled. In that case, the modalities for HH.CatchRegistration should be :
>
>     * "Complete" : SL record is expected for both Landings and
>       Discards fractions . If there is no SL record it is a true 0 value
>     * "Landings" : SL record is expected only for the landed fraction.
>       For this fraction, if there is no SL record, it is a true 0. For
>       the "Discards", no SL record is expected as it has not been
>       registered.
>     * "Discards" : SL record is expected only for the discarded
>       fraction. For this fraction, if there is no SL record, it is a
>       true 0. For the "Landings", no SL record is expected as it has
>       not been registered.
>     * "None" : there is no SL record.
>
> I understand my first proposal (message above) did not separate the 
> information by fraction and was incomplete for raising and calculating 
> the precision. I took the necessary time to ascertain that this 
> proposal is complete and contains all we need.

> Dear All

> Sorry for the delay in response.
>  
> I see; we have different meanings of "partial" in 
> HH.CatchRegistration. Both are valid. A partial registered catch can 
> be registration of only a few selected species or it can be 
> registration of only the discarded or the retained part. So how is 
> this modelled in the best way?
> First question is: What level does this belong to? Is it on TR or on 
> HH? We used to have the partial (species) on TR level coded as 
> sampling type "P". This is good enough for Denmark because we do not 
> have differences between stations on the same trip. What about partial 
> (D/L)? Can this be lifted up to TR-level or do we need to have the 
> possibility to have differences in the partial sampling between FO's? 
> The "None" value in HH.CatchRegistration certainly belong to HH-level 
> an cannot be lifted to TR.
> Next question is: Is this /one/ piece of information, that should be 
> placed in just one field or should it be split into more fields? If it 
> was placed in two fields and one of them had the value "None" and the 
> other "Complete" or "Partial", then it would be self contradicting. On 
> the other hand we might have to add codes for the four possible 
> combinations of partial if we had just one field (e.g. "Partial 
> (Landings all species)" / "Partial (Landings few species)" / "Partial 
> (Discard all species)" / "Partial (Discard few species)"), so the 
> informations are related and intermingled but not one. So we can not 
> answer the question by yes or no. What arguments can then be looked 
> upon? Simplicity for the uploader and for the consumer are important 
> to prevent mistakes. I think both would prefer one field and then 
> limiting the number of sea-sampling codes in the TR.SamplingType to 
> one. The data consumer would then have to look at just one field for 
> inferring zero-catches, no matter if it for discard rates or CPUE. I 
> think both uploader and consumer would find it strange to have 
> "partial (species)" inside TR.SamplingType and then on the same time 
> have Partial (Landings) and Partial (Discard) in a separate field even 
> though one can argue for this.
> Both informations are really a part of the sampling-scheme, that is 
> being pulled out of TR.SamplingType. TR.SamplingType is a composite 
> field, so it is healthy to pull out information in a 
> more DB-normalized and structured way. On the other hand, it does 
> complicate the format if we make too many fields for describing the 
> protocol. However it is necessary if it is on a lower level (HH vs.. TR).
>  
> So I suggest that we have this information in the same field and since 
> the code "None" can not be placed on TR-level, then it should be on HH:
>
>    1. One new field HH.CatchRegistration on HH, remove "P" from
>       TR.SamplingTypes.
>
> Codes for HH.CatchRegistration:
>  
> # "Complete" : SL record is expected for both Landings and Discards 
> fractions for all species. If there is no SL record it is a true 0 value.
> # "Partial (Landings all species)" : SL record is expected for all 
> species, but only for the landed fraction. For this fraction, if there 
> is no SL record, it is a true 0. For the "Discards", no SL record is 
> expected as it has not been registered. 
> # "Partial (Landings some species)" : SL record is expected only for a 
> subset of species and only for the landed fraction. For this fraction, 
> if there is no SL record, then it is not known if it is a true 0. For 
> the "Discards", no SL record is expected as it has not been registered.
> # "Partial (Discards all species)" : SL record is expected for all 
> species but only for the discarded fraction. For this fraction, if 
> there is no SL record, it is a true 0. For the "Landings", no SL 
> record is expected as it has not been registered.
> # "None" : there is no SL record.
>  
> I think that the last possible code ("Partial (Discards some 
> species)") is so theoretical, that we do not have to include it. But 
> it is easily added if needed.
>  
>  
> The alternative options are:
> 2. Two new field HH.CatchRegistrationSpecies and 
> HH.CatchRegistrationDisLan (bad name) on HH, remove "P" from 
> TR.SamplingTypes.
> 3. One new field HH.CatchRegistration (for D/L) on HH, keep "P" from 
> TR.SamplingTypes for species.
>  
> It should be noted that option 2, can be implemented on the DB level 
> even if option 1 is used for the DEF. Option 3 is the option from 
> Joel's previous mail. 
>  
>

> free to pick a solution and then I'll distribute a new version of the 
> DEF-doc.

Hi Teunis an all,

I think we get the whole picture now, it remains only a semantic and 
presentation issue.

It's one hour I'm reflecting/struggling on this now and I wonder if we 
have to distinguish the list of species or not in COST DEF?? I mean, 
it's never "All" species, and a fixed list can be very extensive (app. 
XII + app. XIII of the current DCR is fixed)!! how do we handle this 
information??

My new angle of view on this is  that COST function takes the 
information available as "All" valid for working and raising (at the TR 
level). It is then up to the users to filter their own DB to get the 
correct information to pass on to COST, depending on the demand. This 
would ease up our work considerably and it should be very clear in the 
description of the DEF.

My proposal is now to remove "P" from TR.SamplingType and stick to my 
proposal for HH.CatchRegistration.
