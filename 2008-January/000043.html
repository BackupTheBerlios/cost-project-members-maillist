Teunis Jansen wrote:
> Dear All COST developers working with data input classes/methods
>  
> During programming of the upload and validation in FishFrame 5.0, we 
> have found a deviation from the design rules in the exchange format. 
> This deviation makes problems for our generic coding, so we have to 
> make ugly hacks.
> The design rule is that key fields are inherited from parent data 
> types to child data types and appear there in the same order as in the 
> parent type. So CA record fields should inherit the TR key fields. It 
> does, but I placed /Quarter/ and /Month/ directly after /Year/ to keep 
> the time information gathered, but this was a mistake. TR key fields 
> ends after /Trip number./ Next field is then/ Station no, /the key 
> field for HH. I therefore suggest that we move /Quarter/ and /Month/ 
> to directly after /Station no./
> Both fields (/Quarter/ and /Month/) are key fields, so they should 
> also have an asterisk marking - I just found out that I had forgot 
> putting asterisk on them in the document.
>  
> How would this reordering affect your programming. Would it be OK for you?
>
> Best regards
>
> Teunis
>

Hi Teunis,

The reordering is ok _if_ we agree in keeping the order in COST's class 
definition as it is now. In that case we'll need to change the io 
methods to read the columns in the correct order, which is easy to do.

Best

EJ
