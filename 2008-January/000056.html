>>>> I see; we have different meanings of "partial" in 
>>>> HH.CatchRegistration. Both are valid. A partial registered catch 
>>>> can be registration of only a few selected species or it can be 
>>>> registration of only the discarded or the retained part. So how is 
>>>> this modelled in the best way?
>>>> First question is: What level does this belong to? Is it on TR or 
>>>> on HH? We used to have the partial (species) on TR level coded as 
>>>> sampling type "P". This is good enough for Denmark because we do 
>>>> not have differences between stations on the same trip. What about 
>>>> partial (D/L)? Can this be lifted up to TR-level or do we need to 
>>>> have the possibility to have differences in the partial sampling 
>>>> between FO's? The "None" value in HH.CatchRegistration certainly 
>>>> belong to HH-level an cannot be lifted to TR.
>>>> Next question is: Is this /one/ piece of information, that should 
>>>> be placed in just one field or should it be split into more 
>>>> fields? If it was placed in two fields and one of them had the 
>>>> value "None" and the other "Complete" or "Partial", then it would 
>>>> be self contradicting. On the other hand we might have to add 
>>>> codes for the four possible combinations of partial if we had just 
>>>> one field (e.g.
>>>> "Partial (Landings all species)" / "Partial (Landings few 
>>>> species)" / "Partial (Discard all species)" / "Partial (Discard 
>>>> few species)"), so the informations are related and intermingled 
>>>> but not one. So we can not answer the question by yes or no. What 
>>>> arguments can then be looked upon? Simplicity for the uploader and 
>>>> for the consumer are important to prevent mistakes. I think both 
>>>> would prefer one field and then limiting the number of 
>>>> sea-sampling codes in the TR.SamplingType to one. The data 
>>>> consumer would then have to look at just one field for inferring 
>>>> zero-catches, no matter if it for discard rates or CPUE. I think 
>>>> both uploader and consumer would find it strange to have "partial 
>>>> (species)" inside TR.SamplingType and then on the same time have 
>>>> Partial (Landings) and Partial (Discard) in a separate field even 
>>>> though one can argue for this.
>>>> Both informations are really a part of the sampling-scheme, that 
>>>> is being pulled out of TR.SamplingType. TR.SamplingType is a 
>>>> composite field, so it is healthy to pull out information in a 
>>>> more DB-normalized and structured way. On the other hand, it does 
>>>> complicate the format if we make too many fields for describing 
>>>> the protocol. However it is necessary if it is on a lower level 
>>>> (HH vs.. TR).
>>>> So I suggest that we have this information in the same field and 
>>>> since the code "None" can not be placed on TR-level, then it 
>>>> should be on HH:
>>>>
>>>>    1. One new field HH.CatchRegistration on HH, remove "P" from
>>>>       TR.SamplingTypes.
>>>>
>>>> Codes for HH.CatchRegistration:
>>>> # "Complete" : SL record is expected for both Landings and

>>>> a true 0 value.
>>>> # "Partial (Landings all species)" : SL record is expected for all 
>>>> species, but only for the landed fraction. For this fraction, if

>>>> record is expected as it has not been registered.
>>>> # "Partial (Landings some species)" : SL record is expected only
