Hi Alastair,

my point is to keep it as it is. It is true that some gears have 2 
letters code and some others have 3, but we don't want to redefine the 
EC Regulation. It will be hard enough for the countries to export their 
data in the COST R environment, I don't want to add burden on this.

Question 1 : As David has expressed, the "strsplit" function works very 
well so I encourage the COST developers to use this function to manage 
the metier coding.

Question 2 : It's very true that FAO standards are richer than the EC 
Reg. This is because we have filtered the principal codes for the sake 
of the Regulation, for avoiding panic in the labs!! Anyhow, for COST the 
entire FAO standard applies, and the user may post-stratify gears into a 
group "Other" or "Miscellaneous" to cope with the DCR. Concerning your 
list of gear codes, everything is Ok but GTA?? Don't know what this one 
is? I would say that the risk of proliferating the number of metiers is 
not a COST issue. We take the strata one by one as they are defined by 
the user and proceed to the raising and calculation procedures.

Question  3: In the new code list in annex of the final DEF version (to 
be put very quickly on the website) there is the reference to the FAO 
standard for the gear and the code list for the target assemblages, but 
not anymore a code list for the metier as such. So you must check that 
the first sequence of letters match the FAO standard and the 2nd 
sequence of letters match the target assemblage list.

Hope this clarifies the issue.
